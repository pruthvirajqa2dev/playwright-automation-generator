# ADR-002 — Generated Frameworks Are Independent Repositories

| Field        | Value      |
| ------------ | ---------- |
| **Status**   | Accepted   |
| **Date**     | 2026-06-01 |
| **Deciders** | Core team  |

---

## Decision

Each framework generated by pw-gen is a **self-contained, independent project**.
It has no runtime dependency on pw-gen or on any shared generator library.
After generation, the generated project owns its own dependencies, configuration,
and code.

---

## Context

When building a code generator, there is a fundamental choice about the
relationship between the generator and the generated output:

**Model A — Runtime dependency (shared library):**
The generated project imports a shared library maintained by the generator team.
Upgrades propagate automatically via version bumps.

**Model B — Independent generation (code ownership):**
The generator produces standalone code. The generated project owns every file.
Upgrades require re-generation or manual adoption.

pw-gen uses Model B.

---

## Alternatives Considered

### Model A — Shared runtime library

The generated framework could import a `@pw-gen/core` package that provides
`BasePage`, logger, fixture extensions, and other shared utilities. Teams would
upgrade by bumping the package version.

**Why rejected:**

1. **Constrains evolution.** When `BasePage` is a shared library, every change
   must be backward-compatible across all consumers. Teams cannot customise
   `BasePage` for their application without forking the library.

2. **Hidden coupling.** Test code depends on a package whose internals the
   automation engineer cannot easily read or debug. When the library has a bug,
   the fix is blocked by the library team's release cycle.

3. **Forced uniformity.** Real applications vary. An invoicing system and an
   email client have different wait patterns, different loading indicators,
   different authentication flows. A shared library that serves both becomes
   either overly generic or overly large.

4. **Versioning complexity.** Managing which version of `@pw-gen/core` each
   generated project uses, and coordinating upgrades across a portfolio of
   projects, introduces significant operational overhead.

---

## Reasoning

Generated frameworks are independent because:

1. **Teams own their code.** Every file in the generated framework belongs to
   the automation team. They can read it, modify it, and understand it without
   referring to an external library. There is no black box.

2. **The generator is a starting point, not a runtime contract.** pw-gen
   produces professional-grade starting code that represents current best
   practices. Teams fork that starting point and evolve it for their application.

3. **Long-term maintainability.** When a team's application diverges from the
   generator's assumptions, they change their local code — not a shared library.
   This is the natural evolution of any real project.

4. **Simpler dependency graph.** The generated framework's `package.json` lists
   `@playwright/test`, `winston`, `dotenv` — well-known, stable packages. There
   is no pw-gen package to maintain alongside the generator.

---

## Consequences

**Positive:**

- Generated frameworks are fully portable — no generator infrastructure required to run them.
- Teams can customise any generated file without constraints.
- The generator can change its opinions in future without breaking existing generated projects.
- No library release process required when templates are updated.

**Negative:**

- Improvements to templates do not automatically propagate to existing generated projects. Teams that want new capabilities must re-generate (replacing files they haven't customised) or manually port the changes.
- There is no enforced consistency between generated projects over time — each project diverges as it evolves.

---

## Future Considerations

- An **upgrade command** (`pw-gen upgrade`) could offer a structured migration path — showing diffs between the current generated files and what the latest templates would produce, allowing teams to selectively adopt updates.
- A **drift detector** could compare a generated project against the current template baseline and highlight files that have diverged significantly.
