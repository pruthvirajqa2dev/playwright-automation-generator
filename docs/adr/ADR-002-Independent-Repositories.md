# ADR-002 — Generated Frameworks Are Independent Repositories

| Field        | Value      |
| ------------ | ---------- |
| **Status**   | Accepted   |
| **Date**     | 2026-06-01 |
| **Deciders** | Core team  |

---

## Context

When building a code generator, there is a fundamental architectural choice about
the long-term relationship between the generator and the code it produces.

The choice determines: who owns the generated code, how teams upgrade it, how it
can be customised, and what happens when the generator's assumptions diverge from
the team's application requirements.

---

## Decision

Each framework generated by pw-gen is a **self-contained, independent project**.
It has no runtime dependency on pw-gen or on any shared generator library. After
generation, the generated project owns all of its dependencies, configuration,
and code outright.

The generator is a one-time scaffolding tool. It is not a library, not a runtime,
and not a framework. It produces professional-grade starting code — not a managed
dependency.

---

## Alternatives Considered

### Model A — Runtime Dependency (Shared Library)

The generated project imports a shared `@pw-gen/core` package providing `BasePage`,
logger, fixture extensions, and other shared utilities. Upgrades propagate via
package version bumps.

**Why rejected:**

1. **Constrains evolution.** When `BasePage` is a shared library, every change must
   be backward-compatible across all consumers. Teams cannot customise `BasePage` for
   their application without forking the library or submitting a pull request to a
   shared package.

2. **Hidden coupling.** Test code depends on a package whose internals the automation
   engineer cannot easily read or modify. Debugging requires understanding the
   library's internals in addition to the test's own logic.

3. **Forced uniformity at the wrong level.** Real applications vary. An invoicing
   system and a customer portal have different wait patterns, different loading
   indicators, and different authentication flows. A shared library that serves both
   becomes either overly generic or bloated with escape hatches.

4. **Version management overhead.** Coordinating which version of `@pw-gen/core`
   each generated project uses, handling breaking changes across a portfolio, and
   managing upgrade timing across teams adds significant operational overhead.

5. **CI fragility.** A breaking change in `@pw-gen/core` can break all generated
   projects simultaneously unless each is pinned — which defeats the purpose of
   a shared library.

### Model B — Generation-Time Only (Selected)

The generator produces standalone code. The generated project owns every file and
every dependency. Upgrades require opt-in re-generation or manual adoption.

---

## Rationale

1. **Teams own their code.** Every file in the generated framework belongs to the
   automation team. They can read it, modify it, and understand it without referring
   to an external library. There is no black box.

2. **The generator is a starting point, not a runtime contract.** pw-gen produces
   professional-grade starting code representing current best practices. Teams
   fork that starting point and evolve it for their application.

3. **Long-term maintainability.** When a team's application diverges from the
   generator's assumptions, they modify their local code — not a shared library
   that would affect every other consumer.

4. **Alignment with the Playwright model.** Playwright itself does not provide a
   framework layer — it provides an API. pw-gen's job is to generate a sensible,
   opinionated way of using that API. Wrapping it in a runtime library would add a
   layer of abstraction that obscures Playwright rather than teaching it.

5. **Simple dependency graph.** A generated project depends on Playwright,
   TypeScript, Winston, and a few standard utilities. No custom generator packages.
   The dependency list is immediately legible to any new engineer on the project.

---

## Consequences

**Positive:**

- Generated frameworks are understandable in isolation — no generator knowledge required to work in them
- Teams can deviate from templates freely without worrying about merge conflicts with upstream
- The generator can evolve its templates without backward compatibility constraints
- No single point of failure (a bad library release cannot break all generated projects)

**Negative:**

- Improvements to templates do not automatically propagate to existing projects
- Teams must choose to upgrade — passive benefit is not possible
- The `pw-gen upgrade` command (planned for v0.7) mitigates this by making upgrade adoption explicit and visible

---

## Upgrade Path

Because generated frameworks are independent, the upgrade path is explicit:

- **Re-generation:** Regenerate the framework with updated templates and merge changes
- **Selective adoption:** Manually copy specific updated template outputs into the existing project
- **pw-gen upgrade (v0.7):** The planned upgrade command will diff the current project files against current template output and present a guided adoption flow

The deliberate opt-in nature of upgrades means teams control when and how they adopt changes, rather than being forced into upgrades by a dependency graph.

---

## Future Considerations

- The `pw-gen upgrade` command (v0.7) must be designed with the independent repository model in mind. It cannot assume files are unmodified — teams will have made legitimate changes post-generation.
- Per-file generation metadata (a `generated-at` comment or `.pw-gen-manifest.json`) would enable the upgrade command to identify which files have been modified post-generation.
